20년 차 시니어 풀스택 개발자로서, 제시해주신 기획은 **토스(Toss)의 극도의 직관성, 업비트(Upbit)의 실시간성, 트레이딩뷰(TradingView)의 전문성**이 완벽하게 결합된 훌륭한 아키텍처 아웃라인입니다.

실제 엔터프라이즈급 금융 서비스 수준의 사용자 경험(UX)과 성능, 안정성을 담보하기 위해 기획 검토, 핵심 시나리오, 구체적인 기능 흐름을 설계한 **통합 서비스 시나리오 및 테크니컬 마스터 플랜**을 제안합니다.

## 1. 아키텍처 및 데이터 흐름 설계

프론트엔드와 백엔드, 실시간 소켓 서버, 외부 금융 API 간의 데이터 흐름 구조입니다.

## 2. 핵심 유저 시나리오 흐름

### Scenario 01: 몰입형 진입 & 인터랙티브 로그인

사용자가 서비스에 접속하여 인증을 완료하고 메인 대시보드에 안착하는 첫인상 단계입니다.

* **Step 1: 최초 진입 (Skeleton & Loading)**
* 사용자가 URL에 접속하면 다크 모드(`bg-slate-950`) 기반의 글래스모피즘(Glassmorphism) 스타일 폼이 배치된 로그인 페이지가 로드됩니다. Font는 `Pretendard`가 적용되어 정갈한 서체를 보여줍니다.


* **Step 2: JWT 기반 보안 인증**
* 아이디와 비밀번호를 입력하면 `Axios`가 백엔드로 인증 요청을 보냅니다. 서버는 `bcrypt`로 암호화된 DB 내 비밀번호를 검증한 후, `Access Token(Json)`과 `Refresh Token(HttpOnly Cookie)`을 발급합니다.


* **Step 3: 시그니처 웰컴 애니메이션 (Framer Motion)**
1. 로그인 버튼을 누르면 폼이 부드럽게 사라지고, 화면 중앙에 **"어서오세요 도일님"** 문구가 `y: 20`에서 `y: 0`, `opacity: 1`로 페이드인됩니다.
2. 1초 후, **"어서오세요"** 단어만 `opacity: 0`과 블러 처리가 되며 서서히 기화하듯 사라집니다.
3. 화면 중앙에 남은 **"도일님"** 텍스트가 대시보드 헤더의 우측 상단 좌표를 향해 곡선을 그리며 축소 이동(`layoutId` 기반 공유 레이아웃 애니메이션)합니다.
4. 우측 상단에 안착하는 순간, 자연스럽게 **"도일님의 주식 현황"** 텍스트로 변형(Morphing)되며 전체 대시보드 UI가 스켈레톤 상태에서 실제 데이터 스냅샷으로 트랜지션됩니다.



---

### Scenario 02: 자산 탐색 및 레이아웃 다이내믹 시프트

사용자가 자신의 포트폴리오 상태를 한눈에 파악하고, 특정 종목을 심층 분석하는 핵심 UX 단계입니다.

* **Step 1: 요약 카드(Summary Cards) 탐색**
* 상단에 `TOTAL BUY`, `TOTAL ASSET`, `TOTAL RETURN`, `TOTAL PROFIT` 4개의 카드가 배치됩니다.
* 총 수익률이 플러스(+)면 금융권 표준인 빨간색(`text-red-500`)과 미세한 붉은 네온 글로우 효과가, 마이너스(-)면 파란색(`text-blue-500`)이 카드 보더와 텍스트에 반영됩니다. 마우스 오버 시 `scale: 1.02` 및 그림자가 확장됩니다.


* **Step 2: 레이아웃 다이내믹 트랜지션 (45% vs 55% 분할)**
* 초기에는 `holdings` DB에서 가져온 종목 테이블이 중앙 전체(`max-w-7xl`)를 넉넉하게 차지하고 있습니다.
* 사용자가 테이블의 '삼성전자' 행(Row)을 클릭하는 순간, `Zustand` 스토어의 `selectedStock` 상태가 업데이트됩니다.
* 동시에 CSS Flex/Grid의 `transition-all duration-500 ease-in-out`에 의해 테이블 영역이 좌측 45%로 부드럽게 축소되고, 숨겨져 있던 **우측 55%의 ApexCharts 차트 패널**이 `x: 50`, `opacity: 0` 상태에서 제자리로 미끄러져 들어옵니다.


* **Step 3: 차트 인터랙션 및 실시간 호가 연동**
* 우측 패널 상단에는 실시간으로 변하는 삼성전자의 현재가와 등락률이 표시됩니다.
* 사용자가 1일, 1주, 1개월, 1년 탭을 클릭할 때마다 `TanStack Query`가 백엔드의 `/market/chart?code=005930&range=1M` API를 호출하며, ApexCharts의 캔들스틱 차트가 부드러운 드로잉 애니메이션과 함께 갱신됩니다.

### Scenario 03: 자산 추가 및 스마트 자동완성

보유하고 있는 새로운 주식을 포트폴리오에 등록하는 과정입니다. 데이터 무결성과 예외 처리가 핵심입니다.

* **Step 1: Backdrop Blur 모달 오픈**
* 보유 종목이 없거나 우측 상단의 추가 버튼을 누르면, 뒷배경이 뽀얗게 흐려지는 `backdrop-blur-md bg-black/40` 효과와 함께 추가 모달창이 상단에서 아래로 떨어지듯 가볍게 바운스되며 등장합니다.


* **Step 2: 디바운싱(Debouncing) 기반 종목 검색**
* 사용자가 '삼성'을 타이핑하면 프론트엔드는 300ms 디바운스 타이머를 가동하여 불필요한 API 요청을 방지한 후 백엔드에 검색을 요청합니다.
* 백엔드는 내부 마켓 메타데이터 DB에서 검색하여 `[{ code: '005930', name: '삼성전자' }, ...]` 리스트를 반환하고, 입력창 하단에 드롭다운 형태로 자동완성 목록을 출력합니다.


* **Step 3: 거래 등록 및 실시간 스트리밍 구독**
* 사용자가 '삼성전자'를 선택하면 종목명 칸은 `readonly` 상태로 고정되고, '005930'이라는 종목 코드가 내부 상태에 바인딩됩니다.
* 평균 구매가(예: 72,000원)와 수량(예: 10주)을 입력하고 [저장]을 누르면, 백엔드의 `POST /stocks` API를 거쳐 PostgreSQL의 `holdings` 테이블에 인서트되고, 동시에 `transactions` 테이블에 `type: 'BUY'` 기록이 생성됩니다.
* 저장 완료 즉시 `Socket.io` 클라이언트는 서버에 `socket.emit('subscribe', '005930')` 메시지를 보내 해당 종목의 실시간 가격 푸시 스트림에 가입합니다.

### Scenario 04: 부분/전량 매도(판매) 및 실시간 실현 손익 정산

보유 주식을 시장가로 처분하여 모의 투자 수익을 확정 짓는 시나리오입니다.

* **Step 1: 컨텍스트 드롭다운 메뉴**
* 테이블 우측 끝 `OPTION(세로 점 3개)` 아이콘을 누르면 미니 팝오버 메뉴가 나타납니다. [판매]를 클릭하면 판매 모달이 활성화됩니다.


* **Step 2: 검증 및 한도 제한 인터랙션**
* 모달 내부에는 "현재 최대 10주 보유 중"이라는 가이드 텍스트가 노출됩니다. 사용자가 보유 수량을 초과하는 '12주'를 입력하면 [저장] 버튼이 비활성화되며 "보유 수량을 초과하여 판매할 수 없습니다."라는 경고가 붉은색 글씨로 표시됩니다.


* **Step 3: 잔고 정산 및 청산 알고리즘**
* 사용자가 '5주'를 입력하고 [판매 확인]을 누르면 `POST /stocks/sell` API가 가동됩니다.
* **백엔드 트랜잭션 격리 수준(Isolation Level) 제어:**
1. 현재 서버 메모리 캐시에 존재하는 삼성전자의 최신 가격(예: 75,000원)을 가져옵니다.
2. `holdings` 테이블에서 해당 유저의 삼성전자 주식을 조회합니다.
3. 실현 손익을 계산합니다: $\text{Profit} = (75,000 - 72,000) \times 5 = 15,000\text{원}$.
4. `transactions` 테이블에 `type: 'SELL'`, `price: 75000`, `quantity: 5`, `profit: 15000`으로 로그를 남깁니다.
5. `holdings` 테이블의 기존 수량을 10주에서 5주로 `PATCH` 업데이트합니다. 만약 전량 매도되어 수량이 0이 되면 해당 행을 `DELETE`합니다.


* 완료되면 프론트엔드의 `TanStack Query` 데이터가 `invalidateQueries` 처리가 되어 상단 Summary Cards의 자산 현황과 테이블 정보가 일제히 재계산되며 실시간으로 번쩍이는 시각 효과를 줍니다.

## 3. 기술적 예외 처리 및 엣지 케이스 시나리오

개발 중에 반드시 직면하게 될 예외 상황들과 그에 대한 아키텍처적 해결책입니다.

### Case 01: 외부 금융 API 레이트 리밋(Rate Limit) 및 장애 발생

* **상황:** 한국투자증권이나 Yahoo Finance 같은 외부 주가 API가 점검에 들어가거나 초당 호출 제한을 초과하여 에러를 뱉는 경우.
* **해결책 (서버 사이드 캐싱 체인):** * `node-cron`이 1분마다 외부 API에서 주가를 긁어와 서버 내부 메모리(또는 Redis/로컬 객체) 캐시를 업데이트합니다.
* 만약 외부 API가 `500 Error`를 반환하면 백엔드는 즉시 에러를 뿜지 않고, 가장 최근에 성공했던 캐시 데이터(Stale Data)를 계속 유지하며 소켓으로 던져줍니다. 시스템은 멈추지 않고 "현재 오프라인 시세로 표시 중입니다"라는 가벼운 토스트 알림만 사용자 스크린에 띄웁니다.



### Case 02: 동시성 이슈 및 소켓 커넥션 폭발 (Scalability)

* **상황:** 수많은 유저가 동시에 접속하여 수십 개의 종목을 띄워놓았을 때 소켓 서버 인스턴스의 CPU가 치솟는 현상.
* **해결책 (룸 기반 브로드캐스팅):**
* 모든 유저에게 개별적으로 데이터를 보내는 방식은 $O(N)$의 부하를 줍니다. 대신 종목 코드별로 Socket.io의 `Room`을 생성합니다(`socket.join('ROOM_005930')`).
* 백엔드 크론탭이 1분마다 혹은 주기적으로 가격을 갱신하면, `io.to('ROOM_005930').emit('price_update', data)` 형태로 해당 종목을 보고 있는 유저 그룹에게만 단 한 번의 브로드캐스트로 시세를 전달하여 부하를 $O(1)$에 가깝게 줄입니다.



### Case 03: 모바일 디바이스 환경에서의 데이터 폭탄 및 레이아웃 붕괴

* **상황:** 3G/LTE 환경의 모바일 유저가 접속했을 때, 거대한 테이블과 초당 수십 번 변하는 실시간 소켓 이벤트로 인해 브라우저가 버벅거리는 현상.
* **해결책 (반응형 다운그레이드 및 쓰로틀링):**
* **UI 변경:** 모바일 스크린(`@media (max-w: 768px)`) 환경에서는 가로로 긴 테이블을 제거하고 토스 스타일의 **세로 스크롤 카드형 피드(Feed) UI**로 전면 전환합니다. 행 클릭 시 우측 분할 구조 대신 하단에서 위로 스무스하게 올라오는 **바텀 시트(Bottom Sheet) Drawer** 형태로 차트가 렌더링됩니다.
* **데이터 최적화:** 모바일 디바이스가 백그라운드로 전환되면(유저가 앱을 내리거나 다른 탭을 볼 때) 소켓 연결을 일시적으로 파크(Disconnect/Sleep) 상태로 돌려 배터리와 데이터 소모를 방지하고, 다시 포커스될 때 `TanStack Query`로 스냅샷을 재조회(`refocusOnWindow`)합니다.
